gm-crypto-rs

A pure-Rust SDK for the Chinese national cryptography standards — SM2, SM3, SM4 — with constant-time-designed secret-handling paths guarded by an in-CI leak-regression harness.

{% include project-summary.html %}

What it is

gm-crypto-rs implements GB/T 32905 (SM3 hash), GB/T 32918 (SM2 public-key sign / verify / encrypt / decrypt), and GB/T 32907 (SM4 block cipher) in pure Rust. The crate graph is no_std + alloc, builds on wasm32-unknown-unknown, and ships RustCrypto-trait fits for digest, mac, and cipher. A runnable consumer demo — hash / sign / verify over SM3 and SM2 — lives at gm-crypto-rs-demo.

What is shipped

v0.5 → v0.7 was a throughput-then-API arc: a bitsliced SIMD S-box (AVX2 sbox_x32, NEON sbox_x16) with a CBC-decrypt fanout, then the first user-callable cipher modes (batch APIs, single-shot and streaming SM4-CTR) on top.

v0.8 → v0.11 was an AEAD arc. v0.8 landed the AEAD core — single-shot SM4-GCM and SM4-CCM, plus a constant-time GHASH primitive (CLMUL on x86_64, PMULL on aarch64, software Karatsuba fallback). v0.9–v0.10 added GCM tag-length parameterization, incremental-input buffered GCM, and the single-shot then streaming AEAD C FFI (reachable from C / C++ / Go / Zig / Python). v0.11 modernized the opt-in RustCrypto trait fit to digest 0.11 / cipher 0.5 — default-features users unaffected, every output byte-identical against the KAT suite and gmssl 3.1.1 interop.

v0.12 added SM4-XTS — a tweakable block-cipher mode for disk/sector encryption (GB/T 17964-2021) with full ciphertext stealing, byte-identical to OpenSSL SM4-XTS (xts_standard=GB), behind the opt-in sm4-xts feature.

What's different about it

Other Rust SM2 crates exist — RustCrypto's sm2 is the most established, and it aims for constant-time-designed secret-dependent operations. The differentiator here isn't design intent in isolation. It's an in-CI regression gate that catches a timing leak the moment it shows up, instead of relying on review to assert its absence.

A dudect-bencher harness exercises eighteen secret-touching paths on every CI run. The core set is gated at |τ| < 0.20 — a regression fails the build: SM2 sign, SM2 decrypt, SM4 key schedule, SM4 encrypt (default linear-scan and bitsliced SIMD S-box), SM4-CTR, CBC-decrypt fanout, the single-shot SM4-GCM / SM4-CCM decrypt paths, SM4-XTS decrypt, HMAC-SM3, and encrypted-PKCS#8 decrypt. The field-inversion diagnostics (Fn::invert / Fp::invert), the k-class sign, and the buffered-GCM path run as telemetry on the PR budget (the k-class sign gates at 0.25 on the deeper nightly run). The harness reports detection events — low |τ| means the test couldn't detect a leak with the budget given, not that no leak exists. Language taken directly from dudect-bencher's own docs.

  • #![forbid(unsafe_code)] on gmcrypto-core.
  • All SIMD unsafe quarantined to the sibling crate gmcrypto-simd, opt-in via the sm4-bitsliced-simd feature.
  • Constant-time-designed arithmetic via subtle and crypto-bigint 0.7.
  • Secret material zeroized on drop via zeroize.

What it isn't

  • Not a TLS/TLCP implementation.
  • Not SM9, ZUC, or post-quantum.
  • Not an HSM / SDF / SKF integration.
  • Not a certified cryptographic module.
  • Not constant-time on CPUs with data-dependent multiply latencies (some older x86, some embedded).
