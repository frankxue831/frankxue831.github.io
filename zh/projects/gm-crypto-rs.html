是什么

gm-crypto-rs 用纯 Rust 实现了 GB/T 32905（SM3 哈希）、 GB/T 32918（SM2 公钥签名 / 验签 / 加密 / 解密）、 GB/T 32907（SM4 分组密码）。 整个 crate 依赖图是 no_std + alloc， 可构建到 wasm32-unknown-unknown， 也给出了 RustCrypto 的 digest / mac / cipher trait 适配。 可直接运行的示例（对 SM3 和 SM2 做 hash / sign / verify）放在 gm-crypto-rs-demo。

要解决的问题

Rust 里已经有 SM2 / SM3 / SM4 的实现，写得好的那些在设计上 就奔着秘密相关操作的常量时间去。但「设计意图」只在评审 那一刻成立，之后会悄悄松动——一次重构、一条新的快路径、一次依赖 升级，时序泄漏就可能溜回来，而没有任何东西拦得住。这个 SDK 面对 的恰好是一组让这种松动很容易发生的约束：字节级的 GB/T 一致性、 no_std / wasm 的核心、必须在历次发布之间都保持按常量 时间设计的涉密路径，以及一层把这些同样的路径暴露给非 Rust 调用方 （C、C++、Go、Zig、Python）的 C ABI。它真正围绕的问题不是「今天 是不是常量时间」，而是「靠什么让它在每次提交后都还是常量 时间」。

约束与关键决策

持续验证，而不是一次性断言。 把常量时间当成 CI 每次都要复核的性质，而不是评审签字认可的一句 结论。一套 dudect-bencher harness 在每次 CI 都会跑过 18 个涉及秘密的代码路径；其中核心一组是带门禁的，门槛 |τ| < 0.20——一旦回归就让构建失败，而不是 等人去发现。harness 报告的是检测事件：|τ| 低， 只意味着在给定预算下没检测到泄漏，并不能证明不存在泄漏 （这段措辞直接借用 dudect-bencher 自己的文档）。 代价：每个 PR 都要付出这份测量预算，而统计也永远 没法把「没检测到」变成「不存在」。

核心路径上门禁，其余只做观测。 不是每条路径都塞得进一个 PR 的时间预算，所以 harness 把它们分了层。 带门禁（构建会失败）：SM2 签名与解密、SM4 密钥扩展、 SM4 加密（默认线性扫描和位切片 SIMD S-box 两种）、SM4-CTR、 CBC 解密扇出、单次 SM4-GCM / SM4-CCM 解密、SM4-XTS 解密、 HMAC-SM3、加密 PKCS#8 解密。只做遥测：域求逆诊断、 k-class 签名（在更深的 nightly 上以 0.25 门禁）、 以及缓冲式 GCM。代价：遥测路径只是盯着看、并不 强制，而且这套两层划分本身也得长期维护。

核心保持安全，SIMD 选择性开启。 gmcrypto-core 开了 #![forbid(unsafe_code)]。 位切片 SIMD S-box 需要 unsafe，于是把它整段隔离到 单独的 gmcrypto-simd crate，通过 sm4-bitsliced-simd 特性按需启用。代价： 快路径不是默认——你得主动开 SIMD，开箱构建跑的是更慢的线性扫描 S-box。

按常量时间设计的算术，但把前提讲清楚。 秘密相关的算术建立在 subtle 和 crypto-bigint 之上，秘密材料在 Drop 时由 zeroize 清零。 设计上奔着常量时间去，但没法在所有地方都做到。代价： 在乘法延迟与数据相关的 CPU 上（部分老 x86、部分嵌入式），这些时序 性质并不成立——这点明写在「它不是什么」里，而不是藏起来。

先发核心，再发 FFI——没有变化的版本就不发。 密码模式按 核心在 vN / FFI 在 vN+1 的节奏推进： SM4-XTS 核心在 v0.12、它的 C ABI 在 v0.13；面向磁盘的多扇区原地 辅助在 v0.15、它的 FFI 在 v0.16。当某一轮没有改变任何输出字节—— v0.14 是一次 cargo-fuzz 解析器模糊测试，零崩溃——它就 作为保障性工作合入、永不发布，所以 crates.io 跳过了 0.14.0。代价：版本号上留了一处需要 解释的缺口，而 C ABI 又把每个模式的测试和维护摊到三个已发布的 crate 上，工作量翻了一倍。

证据

每一处输出都做了逐字节比对：对标准 KAT 向量、对 gmssl 3.1.1 的 互操作、对 OpenSSL 3.x SM4-XTS （xts_standard=GB）的可调模式。上面那套泄漏回归门禁 在每次提交的 CI 里都会跑，核心带着 #![forbid(unsafe_code)]。v0.14 那一轮加了一套 cargo-fuzz（libFuzzer）harness——16 个目标覆盖不可信 输入的解码 / 解密路径——作为 nightly 扫描跑。这些都可以查证：源码、 CI 工作流和 dudect harness 都是公开的，连同已发布的 gmcrypto-core crate、它的 docs.rs 文档， 以及可运行的示例。

下一步
候选范围还没敲定——RustCrypto aead 0.6 trait 适配（先搁着，等上游从 0.6.0-rc 转正；v0.11 已经把它需要的 crypto-common 0.2 那条线铺好了）、AVX-512 sbox_x64，以及流式 / 增量 CCM。

它不是什么

  • 不是 TLS / TLCP 实现。
  • 不包括 SM9、ZUC、后量子算法。
  • 不是 HSM / SDF / SKF 集成。
  • 不是经过认证的密码模块。
  • 在某些“乘法延迟跟数据相关”的 CPU 上（部分老 x86、部分嵌入式）不保证常量时间。

个人项目。与 gmssl 3.1.1、OpenSSL 做过逐字节一致性比对， 但并不隶属于这两个项目，也未获它们或任何标准机构、厂商的背书或认证。
