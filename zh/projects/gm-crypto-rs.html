gm-crypto-rs

国密 SM2、SM3、SM4 的纯 Rust SDK。 关键路径按常量时间设计,CI 里挂了泄漏回归门控守着。

{% include project-summary.html %}

是什么

gm-crypto-rs 用纯 Rust 实现了 GB/T 32905(SM3 哈希)、 GB/T 32918(SM2 公钥签名 / 验签 / 加密 / 解密)、 GB/T 32907(SM4 分组密码)。 整个 crate 图是 no_std + alloc, 能编进 wasm32-unknown-unknown, 也给出了 RustCrypto 的 digest / mac / cipher trait 适配。

已经发布了什么

v0.5 → v0.7 是一条"先提吞吐、再开 API"的线: 先做位切片 SIMD S-box(AVX2 sbox_x32、NEON sbox_x16) 加 CBC 解密 fanout,再在上面做出 第一批用户能直接调用的密码模式(批量分组 API、单次和流式 SM4-CTR)。

v0.8 → v0.11 是一条 AEAD 线。 v0.8 落地 AEAD 核心——单次 SM4-GCM 和 SM4-CCM, 外加一个常量时间的 GHASH 原语(x86_64 上走 CLMUL、aarch64 上走 PMULL、 其余平台回退到软件 Karatsuba)。 v0.9–v0.10 加上 GCM tag 长度参数化、增量输入的缓冲式 GCM, 以及单次到流式的 AEAD C FFI(C / C++ / Go / Zig / Python 都能调)。 v0.11 把可选的 RustCrypto trait 适配迁到 digest 0.11 / cipher 0.5—— 默认特性的用户不受影响,所有输出在 KAT 套件和 gmssl 3.1.1 互操作下逐字节一致。

跟其他实现差在哪儿

Rust 里也有别的 SM 系列实现,RustCrypto 的 sm2 是用得最多的—— 它在设计上就奔着"秘密相关操作按常量时间设计"去的。 这里的差异不在"设计意图"本身, 而在 CI 里多了一道回归门控: 一旦冒出时序泄漏,门控立刻拦住, 不靠人工评审去断言"目前没事"。

每次 CI 跑,dudect-bencher 都会扫过 17 个涉及秘密的代码路径。 其中核心一组是带门控的,门槛 |τ| < 0.20——一旦回归就让构建失败: SM2 签名、SM2 解密、SM4 密钥扩展、SM4 加密 (默认线性扫描和位切片 SIMD S-box 两种都跑)、SM4-CTR、CBC 解密 fanout、 单次 SM4-GCM / SM4-CCM 解密路径、HMAC-SM3、加密 PKCS#8 解密。 域求逆诊断(Fn::invert / Fp::invert)、k-class 签名、 以及缓冲式 GCM 路径在 PR 预算下只做遥测 (k-class 签名在更深的 nightly 上以 0.25 门控)。

harness 报告的是检测事件—— |τ| 低,只意味着这次预算下没检测到泄漏, 并不等于"常量时间"。 这段措辞是直接借用 dudect-bencher 自己的文档。

  • gmcrypto-core 开了 #![forbid(unsafe_code)]。
  • 所有 SIMD 的 unsafe 都隔离到 sibling crate gmcrypto-simd,通过 sm4-bitsliced-simd 特性按需启用。
  • 按常量时间设计的算术用 subtle 配 crypto-bigint 0.7。
  • 秘密材料在 drop 的时候由 zeroize 清零。

下一步

v0.12
SM4-XTS——面向磁盘/扇区加密的可调分组模式(IEEE 1619,推广到 SM4),支持完整的密文挪用,和 OpenSSL SM4-XTS 逐字节一致。计划中,尚未发布。
暂缓
RustCrypto aead 0.6 trait 适配先搁着,等上游从 0.6.0-rc 转正;v0.11 已经把它需要的 crypto-common 0.2 那条线落好了。

它不是什么

  • 不是 TLS / TLCP 实现。
  • 不包括 SM9、ZUC、后量子算法。
  • 不是 HSM / SDF / SKF 集成。
  • 不是经过认证的密码模块。
  • 在某些"乘法延迟跟数据相关"的 CPU 上(部分老 x86、部分嵌入式)不保证常量时间。

个人项目。不隶属、也没有被任何上游密码库、 支付网关、标准机构或厂商背书或认证。
