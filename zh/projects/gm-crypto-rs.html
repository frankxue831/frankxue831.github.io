gm-crypto-rs

国密 SM2、SM3、SM4 的纯 Rust SDK, 关键路径常量时间设计、由 CI 中的泄漏回归门控守护。

是什么

gm-crypto-rs 用纯 Rust 实现 GB/T 32905(SM3 哈希)、 GB/T 32918(SM2 公钥签名 / 验签 / 加密 / 解密)、 GB/T 32907(SM4 分组密码)。 crate 图为 no_std + alloc, 可在 wasm32-unknown-unknown 上构建, 并提供 RustCrypto 的 digest / mac / cipher trait 适配。

有什么不同

社区里已有不错的 Rust SM2 实现——RustCrypto 的 sm2 比较成熟, 其设计意在做到关键路径常量时间。 这里的差异点不在"设计意图"本身, 而在 CI 中的回归门控: 一旦有泄漏出现,门控会立即拦截, 而不是依赖人工评审去断言"目前没有泄漏"。

每次 CI 中,15 个涉及秘密的代码路径都会在 dudect-bencher 下跑一遍, 门槛为 |τ| < 0.20。 覆盖 SM2 签名、SM2 解密、SM4 密钥扩展、SM4 加密 (默认线性扫描与位切片 SIMD S-box 两种都跑)、HMAC-SM3、 加密 PKCS#8 解密,以及 Fn::invert / Fp::invert 的直接诊断。 harness 报告的是检测事件—— |τ| 低只意味着在给定预算下未能检测到泄漏, 并不证明常量时间。 措辞直接借用 dudect-bencher 自身的文档。

安全姿态

  • gmcrypto-core 启用 #![forbid(unsafe_code)]。
  • SIMD unsafe 全部隔离到 sibling crate gmcrypto-simd,通过 sm4-bitsliced-simd 特性按需启用。
  • 常量时间算术依赖 subtle 与 crypto-bigint 0.7。
  • 秘密材料在 drop 时通过 zeroize 清零。

吞吐量从哪儿来

v0.5 → v0.6 是一条 SIMD 演进线。 v0.5.0 搭好脚手架, v0.5.1 加上 AVX2 的 sbox_x8, v0.6.0 把它拓宽到 AVX2 上的 sbox_x32, 并在 aarch64 上加了 NEON 的 sbox_x16, CBC 解密做了 fanout,让单块延迟藏在并行 S-box 后面。 v0.7(开发中)在此之上做出第一批用户可调用的密码模式。

它不是什么

  • 不是 TLS / TLCP 实现。
  • 不包括 SM9、ZUC、后量子算法。
  • 不是 HSM / SDF / SKF 集成。
  • 不是经过认证的密码模块。
  • 在某些有"数据相关乘法延迟"的 CPU 上(部分老 x86、部分嵌入式)不保证常量时间。
